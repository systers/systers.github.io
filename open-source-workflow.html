Open Source Workflow

Below is the process that Systers uses to plan, create, test, and deploy our various projects within GitHub. These are the steps that should be taken whenever contributing to any of our repositories.

Software Development Life Cycle

Requirements Analysis

Project Requirements Doc Create Issues Approved by Mentor/Maintainer

Design

Create Mock-Ups for the Issue Reviewed by Mentor/Maintainer Adjust Mock-Up Based on Feedback Approved by Mentor/Maintainer

Development & Testing (TDD)

Code/PR/Test to Complete the Issue Reviewed by Mentor/Maintainer Adjust Code/PR/Test Based on Feedback Approved by Mentor/Maintainer

Testing

Create Issues that Test the Applications in Production (i.e. User Tests, Feature Tests, etc.) Reviewed by Committer Submit PRs that Contain Documentation on Tests to Suggest Improvements for Next Release Approved by Committer

Maintenance

Create Issues that Fix Errors/Improve Impelentation/Upgrade the Current Language Approved by Mentor/Maintainer/Committer

Before Merging a Pull Request

  1. Check whether the PR was sent to the correct branch.
  2. Check whether the PR has unnecessary merge commits. If it does squash and merge, or ask the student to do it again by rebasing (removing the merge commit). My recommendation is for a first timer tell him/her that the recommended way is rebasing and do a squash and merge. So that they wonâ€™t get discouraged and go away. For a regular if they keep doing this do not merge until they send a PR with merge commits removed.
  3. Even if the student started with a rebased branch, they may encounter merge conflicts. In that case ask the student to fix the merge conflict. When they are fixing this again the merge is the easiest way at that point so merge commits can be visible. In that case ask the student to squash them or you can squash them when merging as instructed.

Squash Commits

Squash Commits means when you have a list of commit and you want to make it one commit or you may get asked from any mentor to squash the commits, so here are the steps you can follow:

When you select the Squash and merge option on a pull request on GitHub, the pull request's commits are squashed into a single commit. Instead of seeing all of a contributor's individual commits from a topic branch, the commits are combined into one commit and merged into the default branch. (from Github Documentation)
  1. git log
    2. You get the log of last commits on the branch, which you are working!
  2. git branch
    3. You can check, on which branch you are!
  3. git rebase -i master
    4. This command will show up a list of all commits
  4. leave the commit as pick only which you want to be the main commit, and change other Commits as squash which you want to include in the main commit
  5. It will look something like this:
    6. pick fb554f5 commit 1
    squash 2bd1903 commit 2
    squash d987ebf commit 3

Run this command and then you can force push if you already submitted a pull request by git push -f
